# Дизайн-Документ: opencode-scaffold (v6.0 — Финальная Архитектура и Синтез)

## 1. Введение и Стратегия Реворка
`opencode-scaffold` (ранее `claude-scaffold`) — это локальный CLI-оркестратор для OpenCode, предназначенный для развертывания идеальной субагентной среды, архитектуры памяти (Memory Bank) и визуального мониторинга (TUI).

**Стратегия Git-репозитория:** Мы работаем в новой чистой ветке `v2-rework`. Вся устаревшая инфраструктура остается в `main`. Мы будем "подглядывать" в старые директории (например, забирать из них промпты и метрики), но строим новую архитектуру с нуля, чтобы избавиться от легаси-кода.

---

## 2. Глубокий Анализ Агентной Системы: Internal vs External
В предыдущих версиях дизайна мы качались от полностью внешних плагинов (`oh-my-opencode`) к полностью нативным агентам OpenCode (`.opencode/agents/`). Критический анализ документации OpenCode показывает, что **правда в гибридном подходе (Синтезе)**.

### Нативная система (Internal):
*   **Что делает лучше всего:** Идеально описывает *сущности* агентов. Markdown-файлы во `frontmatter` (`.opencode/agents/qa_engineer.md`) позволяют жестко привязать конкретную модель (например, дешевый `GLM-5-Turbo` для линтера, мощный `Gemini-3.1-Pro` для архитектора), задать им System Prompt и выдать конкретные допуски (permissions).
*   **Где ломается:** Нативная система не умеет делать *фоновые retry-циклы*. Если ты просто попросишь нативного субагента написать тесты, и они упадут, он будет писать логи падений прямо в основной контекст диалога, раздувая его и сжигая токены.

### Внешние плагины (External — `oh-my-opencode` / Orchestrators):
*   **Что делает лучше всего:** Выполняет задачи в *изолированных фоновых песочницах* (ultrawork). Оркестратор может заставить субагента сделать 5 попыток починить упавший тест, и если он справится, вернуть в главный диалог **только** финальный результат ("Тесты пройдены, вот отчет"), скрыв весь процесс "мучений" субагента. Это колоссальная экономия контекста.
*   **Синтез (Как мы это строим):**
    1. Утилита `init` генерирует **нативные** Markdown-файлы агентов с нашими "Золотыми промптами" (Architect, QA, Security, Performance).
    2. Утилита устанавливает **внешний плагин** оркестрации (`oh-my-opencode-slim` или `GolemBot`).
    3. Главный агент использует инструмент плагина (например, `delegateTask_Background`), скармливая ему нативного субагента. Это дает нам гибкость нативного конфига и мощь внешней оркестрации.

## 3. Критически важные фичи (Упущенное ранее)

При анализе SOTA-решений для CLI-агентов выявились 3 критических пробела, которые мы обязаны закрыть:

1.  **Динамическое сжатие контекста (Dynamic Context Pruning - DCP):**
    *   Даже с Memory Bank, если агент читает логи огромной ошибки от компилятора, его контекст забивается мусором.
    *   *Решение:* Интеграция плагина **`@tarquinen/opencode-dcp`**. Он автоматически вырезает (prune) из истории старые выводы bash-команд, оставляя только их summary, что продлевает жизнь сессии в 2-3 раза.
2.  **LSP Интеграция (Language Server Protocol):**
    *   Агент тратит время на запуск скриптов, чтобы найти ошибку импорта.
    *   *Решение:* Использование LSP-тулзов (например, `typescript-tools` или `pyright-tools` через MCP). Агент должен иметь возможность вызвать `get_diagnostics()`, чтобы моментально увидеть ошибки линтера в открытом файле, как это делает программист в VS Code.
3.  **Cross-Repository Knowledge (Синхронизация Хаба):**
    *   Хотя Scaffold устанавливается локально, он должен знать о "соседях".
    *   *Решение:* Оставить Read-Only доступ к глобальному индексу `TechCon Hub` (папка `knowledge`), чтобы локальный агент мог прочитать стандарты АО РР или БРИС ЖКХ.

## 4. Архитектура Настоящего E2E-Тестирования (Scaffold Pipeline)
Чтобы гарантировать качество самого `opencode-scaffold` и его субагентов, мы отказываемся от синтетических юнит-тестов (pytest stubs) в пользу **поведенческого (BDD) тестирования реальных сценариев**. 

Папка `tests/e2e/` будет содержать сценарии на базе фреймворков `Cucumber` + `zx` (библиотека Google для написания bash-скриптов на JS/TS).

**Реальные сценарии E2E (Как мы тестируем сам Scaffold):**
*   **Scenario 1 (Bootstrap & Init):** Скрипт создает пустую папку, запускает `npx opencode-scaffold init`, проверяет создание `.opencode/agents/*.md`, `.opencode/config.json`, и правильное копирование 4-х файлов Memory Bank.
*   **Scenario 2 (AST RAG Pipeline):** E2E тест натравливает наш RAG-индексатор (`tree-sitter`) на 1000-строчный Python файл (например, фейковый роутер FastAPI) и валидирует, что на выходе получается ровно 50 строк Markdown-сигнатур без тел функций.
*   **Scenario 3 (TUI IPC Polling):** Поднимается 3 фейковых Node.js процесса, эмулирующих OpenCode-агентов. Они пишут JSON стейты. Инстанс `opencode-tower` (TUI) запускается в headless-режиме, и E2E тест проверяет, что дашборд корректно парсит токены из 3-х источников одновременно.
*   **Scenario 4 (Memory Bank Update Loop):** Запуск реального headless-агента (через CLI), которому дается команда "Создай калькулятор". Тест ждет завершения и проверяет `git diff` файлов `activeContext.md` и `progress.md` — агент **обязан** был их обновить.

## 5. Пошаговый План Имплементации (Roadmap)

### Шаг 1. Фундамент CLI (Node.js)
*   **Инструменты:** `commander`, `inquirer`, TypeScript, `tsup` (для сборки).
*   **Действие:** Создание команды `init`. Она генерирует нативных агентов OpenCode (Markdown-файлы) из захардкоженных "золотых промптов" (Architect, QA, Sentinel) в целевом репозитории. Установка `.opencode/config.json` с включенным веб-поиском.

### Шаг 2. Развертывание Memory Bank
*   **Действие:** Команда `init` также генерирует 5 файлов памяти (`projectbrief`, `activeContext`, `systemContext`, `productContext`, `progress`).
*   Формирование жесткого `OPENCODE.md` (System Prompt), который обязует агента читать и писать в эти файлы.

### Шаг 3. Интеграция Плагинов (Синтез Internal + External)
*   **Действие:** Scaffold автоматически прописывает в `package.json` репозитория зависимости: `oh-my-opencode-slim` (для фонового оркестрирования субагентов) и `@tarquinen/opencode-dcp` (для сжатия контекста bash-выводов).

### Шаг 4. Телеметрия и AST-RAG (Локальный Python/SQLite)
*   **Действие:** Написание скрипта `generate_ast.py` на базе `tree-sitter`.
*   Написание `telemetry_server.js`, который слушает OTLP-трейсы и пишет метрики в `.opencode/telemetry.db`. 

### Шаг 5. Pre-commit Loop & E2E
*   **Действие:** Генерация `.pre-commit-config.yaml` (`ruff`, `mypy`, `eslint`).
*   Написание E2E тестов для самого Scaffold (описано в Секции 4), чтобы мы могли безопасно рефакторить наш код.

### Шаг 6. Control Tower (TUI)
*   **Действие:** Создание команды `opencode-scaffold tower`. Она использует библиотеку `ink` для отрисовки терминальной таблицы. Дашборд парсит `state.json` из всех запущенных сессий (которые мы настроим писать свои стейты в глобальную папку) и выводит их в psmux.

---

## 6. Итог
Новая ветка `v2-rework` позволяет нам отбросить костыли старого Scaffold. Мы строим идеальный гибрид: нативное определение агентов + внешняя оркестрация (`oh-my-opencode`) + жесткое сжатие контекста (`opencode-dcp`) + надежная E2E валидация (Cucumber/zx). Этот оркестратор сделает разработку в `TechCon` дешевле (по токенам) и кратно надежнее.
